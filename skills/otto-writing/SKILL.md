# 技术写作执行规范

## 何时启用
- 用户要写技术博客 / 教程 / 指南 / 方案说明 / 故障复盘。
- 用户目标是“读者看完就能做”，而不是“读完感觉不错”。
- 默认产物是 Markdown 技术文，用于知识库、博客平台或团队文档。

## 路由边界
- 若用户要的是技术内容成稿（Markdown），使用本 Skill。
- 本 Skill 不生成后台、Dashboard、表单工作台或 Web App 原型。

## 核心规则
1. 先给证据，再给解释（代码、命令、日志、输出、数据）。
2. 先举例，再讲原理，不反过来。
3. 开头先说明读者收获：读完能完成什么、能避开什么坑。
4. 每节只解决一个问题，避免一节多目标。
5. 用可验证事实替代形容词，不编造案例、数据、背书。
6. 句子短、段落紧，删掉空泛过渡句与 AI 套话。
7. 每节结尾给“下一步动作”或“完成判定”，保证可执行。

## 流程（Phase）

### Phase 0 — Source
收集四项输入：
- 目标读者
- 成功标准（读完后能做成什么）
- 可用材料（代码、日志、链接、截图）
- 发布介质（博客、知识库、团队文档）

### Phase 1 — Plan
产出一个简版计划（可内联），必须包含四段：
- Brief：读者、目标、语气、信息密度、保留范围
- Outline：章节列表（每节一句目标）
- Theme：内容风格与叙述策略
- Assets：每节证据与素材映射（代码、日志、图表、截图）

### Phase 2 — Checkpoint（必须停）
逐项确认决策，不打包成一个 yes/no：
1. 文章类型（tutorial / explainer / review）
2. 信息密度（高 / 中 / 低）
3. 语气（克制 / 直接 / 叙述）
4. 图表策略（defer / none / user-assets / placeholders）
   - 默认 `defer`：先完成全文，再判断是否需要配图

### Phase 3 — Sample
先产出一节代表性样张，再继续全稿：
- 样张必须包含“证据 → 解释 → 动作”完整链路
- 样张用于校准语气、信息密度与证据粒度
- 样张确认后再进入 Full 阶段

### Phase 4 — Full
按已确认的 Plan 与样张扩展为全稿：
- 按 Outline 顺序逐节完成
- 保证节间衔接，不重复解释同一事实
- 保持证据粒度与术语口径一致

### Phase 4.5 — Visual Decision（Draft-first）
在 Full 稿完成后，逐节做一次配图判断，不预设“必须有图”。

逐节判断：
- 无图时，读者是否仍能准确执行本节动作？
- 图是否能明显降低理解成本（关系、流程、对比、定位）？

决策分级：
- `must-have`：无图会误解、漏步骤或难以复现
- `nice-to-have`：无图可完成，但有图可明显提速理解
- `none`：图不增加有效信息

产出：
- 配图清单（节标题 / 决策分级 / 图的用途 / 素材来源）

### Phase 5 — Final Review
三视角终审：
- Editorial：结构是否推进、是否冗余、是否跑题
- Visual：排版可读性、代码块与列表层级是否清晰
- Technical：命令/代码/结论是否可复现

## 节模板（强制）

### <节标题>
**证据**
```bash
# command / output / log / snippet
```

**解释**
- 发生了什么：
- 为什么：
- 边界与误区：

**动作**
- 现在该做：
- 完成判定：

## 质量闸门（交付前）
- [ ] 是否先证据后解释
- [ ] 是否先例子后原理
- [ ] 开头是否明确读者收获
- [ ] 关键主张是否有证据锚点
- [ ] 是否删除空泛过渡句与营销词
- [ ] 风格是否前后一致（术语、语气、粒度）
- [ ] 每节是否有新增信息
- [ ] 结尾是否给出明确下一步
- [ ] 配图仅用于承载结论（非装饰）
- [ ] 正文是否引用图片并说明看点（图1/图2）
- [ ] 图注是否包含用途、来源、版本或时间
- [ ] 图片是否已脱敏且与正文一致

## 禁用模式
- 空话开场（如“在快速发展的时代……”）
- 无证据的“最佳实践”断言
- 把“为什么重要”单独成段但不给事实
- 仅为互动而提问、但不推进论点

## 输出契约
- 默认输出：Markdown 完整成稿
- 可选附加：证据-结论映射清单（用于审阅追溯）
- 允许最终为 0 图：当 Visual Decision 结论为 `none` 时，不强行配图
