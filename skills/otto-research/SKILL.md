# 技术深度研究

## 何时启用

用户消息中出现以下任一模式：
- "研究/调研/调查/深入分析 一下 X"
- "X 是什么 / 为什么要有 X / X 解决了什么"
- "X 和 Y 对比哪个好"
- "怎么做 X / X 怎么实现"
- "X 的发展历程 / X 是怎么来的"

## 核心原则

1. **信息呈现优先**：先给结论，再给解释，最后给可执行建议。
2. **来源优先**：官方信息 > 大型公开资料站 > 个人博客。同主题下，有剖析有调查的内容 > 纯结论内容。
3. **交叉比对（过程内化）**：关键事实优先做交叉确认，但默认不在正文展示证据分级标签。
4. **聚焦问题**：报告内容按研究问题类型调整侧重，不塞无关信息。
5. **不确定性最小外显**：仅在结论受影响时给一句必要说明，不展开审计叙事。

## 研究问题分型

| 类型 | 核心要回答 | 报告侧重 | 触发示例 |
|---|---|---|---|
| **是什么** | 它是什么、为什么出现、解决了什么 | 概念解释、产生背景、核心机制 | "WebAssembly 是什么" |
| **怎么做** | 如何实现、什么场景适用、有什么坑 | 实现步骤、示例、踩坑提醒 | "Rust 中怎么做异步编程" |
| **为什么** | 问题如何产生的、根因是什么 | 因果链、触发条件、关联因素 | "为什么 React Server Components 要推" |
| **发展历程** | 怎么演变来的、各阶段解决了什么 | 时间线、关键节点、演进逻辑 | "TypeScript 发展历程" |

## 工作流程

### 1) 明确研究目标与边界

接到研究任务后，先判断：

- 问题类型（上表）
- 用户环境中涉及的技术栈/版本（若缺失且影响结论，1-2 问补全）
- 默认范围：近 12 个月优先，除非用户指定

**边界不明确时必须停止，向用户提出澄清问题。**
出现以下任一情况，不得继续拆分或搜索：
- 核心问题有歧义，存在多种合理但截然不同的理解
- 用户指定的范围过于宽泛（如"研究一下 AI"），无法收束为有限子问题
- 子问题之间存在大片重叠，无法划清边界
- 研究问题类型无法归类到分型表中的任意一类

澄清问题应 ≤ 2 个，聚焦在消除歧义或收束范围上。

### 2) 拆分 2-5 个子问题

- 每个子问题必须是可验证的具体问题，边界互不重叠
- 所有子问题合并后应完整覆盖用户关心的范围

拆分后先给用户确认方向再开始搜索。确认时输出：

```
## 研究方向确认

### 核心问题
[一句话]

### 子问题
1. [子问题 1]
2. [子问题 2]
...

### 预期来源类型
[列出计划检索的官方文档、技术社区等]

### 不覆盖
[明确排除的范围]
```

### 3) 多源搜索与收集

**工具限制**：仅使用平台内置搜索工具与可用 MCP 搜索工具。不编写爬虫、不构造 HTTP 请求、不使用 Browser 插件内的脚本抓取能力。

**搜索预算**：
- 整个研究任务：**最多 15 次搜索**
- 每个子问题：**3-5 次搜索**（用满前先评估是否已收集足够）
- 命中高质量一手来源后，该子问题即可停止搜索，不为凑次数继续检索

每个子问题使用 2-3 组关键词覆盖：
- 通用信息查询
- 官方文档/仓库
- 反证查询：bug、limitation、弃用、争议

收集内容整理为"候选主张"，每条包含：
- 一句话主张
- 来源 URL
- 来源类型（官方文档/仓库/媒体/社区/个人）
- 发布日期或版本
- 初步可信度标注

### 4) 网络受限处理

**不重试原则**：官方源不可达（连接超时、DNS 失败、返回错误码）时，直接认定为不可达，不执行重试。

**降级路径**：

| 目标源不可达 | 替代方案 |
|---|---|
| 官方文档/官网 | `web.archive.org` 存档快照；国内技术社区的原文翻译或逐段引用 |
| 官方仓库（GitHub 等） | 国内镜像站（Gitee、清华 tuna）；仓库 README 的翻译或引述 |
| 外网技术媒体 | 国内同类来源（掘金、知乎技术板块）的同主题深度文章 |

**输出规则（信息呈现优先）**：
- 默认不单独输出“网络环境声明”章节。
- 仅在相关结论受影响时，用一句话提示来源受限与影响范围。
- 不输出证据分级标签化措辞。

### 5) 筛选与去重

按两个维度筛选：

**来源可靠性**：
1. 官方文档、官方仓库、标准组织
2. 大型技术社区、知名技术媒体
3. 个人博客、论坛

**内容深度**：
- 有剖析过程、有数据支撑 > 纯结论

**去重规则**：
- 同一事实的转载/二次转述合并为 1 条独立证据
- 同机构不同渠道（官网博客 + 官方社媒）按 1 条处理
- 替代源与其原始官方源按 1 条处理（不因多渠道转载而算多来源）
- 冲突来源保留，标记为「存疑」

### 6) 交叉比对与证据门槛

对关键结论执行内部比对：

- 是否有独立来源支撑？
- 来源之间是否存在冲突？
- 结论的时间/版本是否匹配用户环境？

**冲突时判定优先级**：
- 时序：原始出处 > 后续转述
- 权威性：官方 > 第三方
- 详细度：有完整论证 > 一句话断言

**输出规则**：
- 以上门槛用于研究过程质量控制。
- 默认正文不展示证据分级标签。
- 如存在关键冲突，使用读者友好句式说明“口径不一致”及建议动作。

### 7) 输出报告

按问题类型组织内容，但统一遵循“信息呈现优先”的结构：

```
# [主题] 研究报告
生成日期 | 适用范围

## TL;DR
- 核心结论 1
- 核心结论 2
- 下一步建议

## 核心信息（按子问题）
### 子问题 1
- 结论
- 为什么
- 怎么用

### 子问题 2
- 结论
- 为什么
- 怎么用

## 可执行建议（按优先级）
1. 建议 A：适用条件｜操作｜预期结果
2. 建议 B：适用条件｜操作｜预期结果

## FAQ（高频问题）
Q1: ...
A1: ...

## 延伸阅读
1. [标题](url) — 用途
```

**表达规则（默认）**：
1. 每节首句先给结论。
2. 使用“是什么/为什么/怎么用”三层表达。
3. 标题动作化（如：怎么选、怎么做、怎么避坑）。
4. 默认不使用证据分级标签化措辞。
5. 建议必须包含：适用条件 + 操作 + 预期结果。
6. 内容长度由问题复杂度决定，不设置硬性字数或行数限制。
7. 来源仅作为延伸阅读，不做来源分级展示。

### 8) 本地落盘（默认）

研究结果保存到 `output/research/YYYY-MM-DD-topic/` 目录：
- `report.md` — 完整报告
- `meta.json` — 研究范围、子问题、搜索词、时间戳

## 质量门禁

交付前检查：
- [ ] TL;DR 可独立传达核心结论
- [ ] 每个子问题都包含“结论 + 怎么用”
- [ ] 每条建议都包含“条件 + 操作 + 结果”
- [ ] 术语首次出现有白话解释
- [ ] 正文无审计标签化措辞
- [ ] 延伸阅读与正文强相关，数量由问题复杂度决定
- [ ] 无“无来源断言”
- [ ] 最终审计仅检查：AI 口吻、内容重复

## 快速示例

**输入**："研究一下 WebAssembly 是什么"
1. 判断类型 → "是什么"
2. 拆分子问题 → 定义、产生背景、解决的问题、与相关技术关系
3. 搜索与整理 → 收集关键信息并提炼结论
4. 输出正文 → TL;DR + 核心信息 + 可执行建议 + FAQ
5. 补充延伸阅读 → 按相关性列出可继续阅读的来源
6. 最终审计 → 只检查 AI 口吻与内容重复
