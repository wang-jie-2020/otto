## 架构评审流程

### 1. 当前状态分析
- 审查现有架构
- 识别模式和约定
- 记录技术债
- 评估可扩展性限制

### 2. 需求收集
- 功能性需求
- 非功能性需求（性能、安全、可扩展性）
- 集成点
- 数据流需求

### 3. 设计提案
- 高层架构图
- 组件职责
- 数据模型
- API 契约
- 集成模式

### 4. 权衡分析
对每个设计决策记录：
- **Pros**：收益和优势
- **Cons**：缺点和限制
- **Alternatives**：考虑过的其他选项
- **Decision**：最终选择及理由

## 架构原则

### 1. 模块化与关注点分离
- 单一职责原则
- 高内聚、低耦合
- 组件之间接口清晰
- 独立可部署性

### 2. 可扩展性
- 水平扩展能力
- 尽可能无状态设计
- 高效数据库查询
- 缓存策略
- 负载均衡考虑

### 3. 可维护性
- 清晰的代码组织
- 一致的模式
- 完整文档
- 易于测试
- 易于理解

### 4. 安全
- 纵深防御
- 最小权限原则
- 在边界处进行输入验证
- 默认安全
- 审计轨迹

### 5. 性能
- 高效算法
- 最少网络请求
- 优化数据库查询
- 合适的缓存
- 延迟加载

## 常见模式

### 前端模式
- **组件组合**：用简单组件构建复杂 UI
- **容器/展示组件**：分离数据逻辑和展示
- **自定义 Hooks**：可复用的有状态逻辑
- **Context 管理全局状态**：避免 prop drilling
- **代码分割**：懒加载路由和重型组件

### 后端模式
- **Repository Pattern**：抽象数据访问
- **Service Layer**：分离业务逻辑
- **Middleware Pattern**：请求/响应处理
- **Event-Driven Architecture**：异步操作
- **CQRS**：读写操作分离

### 数据模式
- **规范化数据库**：减少冗余
- **为读性能反规范化**：优化查询
- **Event Sourcing**：审计轨迹和可重放性
- **缓存层**：Redis、CDN
- **最终一致性**：用于分布式系统

## 架构决策记录（ADRs）

对重要架构决策创建 ADR：

```markdown
# ADR-001: 使用 Redis 作为语义搜索向量存储

## Context
需要存储并查询用于语义市场搜索的 1536 维 embedding。

## Decision
使用具备向量搜索能力的 Redis Stack。

## Consequences

### Positive
- 向量相似度搜索快（<10ms）
- 内置 KNN 算法
- 部署简单
- 在 100K 向量以内性能良好

### Negative
- 内存存储（大数据集成本高）
- 无集群时存在单点故障
- 限于余弦相似度

### Alternatives Considered
- **PostgreSQL pgvector**：更慢，但持久化存储
- **Pinecone**：托管服务，成本更高
- **Weaviate**：功能更多，设置更复杂

## Status
Accepted

## Date
2025-01-15
```

## 红旗

警惕这些架构反模式：
- **Big Ball of Mud**：没有清晰结构
- **Golden Hammer**：用同一种方案解决所有问题
- **Premature Optimization**：过早优化
- **Not Invented Here**：拒绝现有方案
- **Analysis Paralysis**：过度规划、构建不足
- **Magic**：行为不清晰、无文档
- **Tight Coupling**：组件过度依赖
- **God Object**：一个类/组件做所有事情

## 项目特定架构（示例）

AI 驱动 SaaS 平台的示例架构：

### 当前架构
- **Frontend**：Next.js 15（Vercel/Cloud Run）
- **Backend**：FastAPI 或 Express（Cloud Run/Railway）
- **Database**：PostgreSQL（Supabase）
- **Cache**：Redis（Upstash/Railway）
- **AI**：Claude API 搭配结构化输出
- **Real-time**：Supabase subscriptions

### 关键设计决策
1. **混合部署**：Vercel（前端）+ Cloud Run（后端），获得最佳性能
2. **AI 集成**：使用 Pydantic/Zod 结构化输出以保证类型安全
3. **实时更新**：使用 Supabase subscriptions 获取实时数据
4. **不可变模式**：使用 spread operators 获得可预测状态
5. **许多小文件**：高内聚、低耦合

### 可扩展性计划
- **10K 用户**：当前架构足够
- **100K 用户**：添加 Redis 集群，静态资源使用 CDN
- **1M 用户**：微服务架构，读写数据库分离
- **10M 用户**：事件驱动架构、分布式缓存、多区域

**记住**：好的架构支持快速开发、轻松维护和有信心的扩展。最好的架构简单、清晰，并遵循既有模式。
